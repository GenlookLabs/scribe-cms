## Added

- `scribe translate` now runs through the Gemini Batch API by default, cutting token costs by 50%. All requests are planned and submitted upfront (grouped per model, chunked for large worklists), then polled together and ingested as each job completes
- Translation runs are resumable: batch jobs and their items are persisted in the SQLite store the moment they are submitted, so quitting during polling loses nothing. Run `scribe translate --resume` (or simply `scribe translate` again) to pick pending jobs back up, ingest finished ones, and skip anything already in flight
- New `--batch` and `--direct` flags choose the mode explicitly; in a terminal, an interactive "Translation mode" prompt offers the choice with batch preselected
- Transient Gemini errors (429, 5xx, network drops) are retried automatically with exponential backoff instead of failing the item

## Changed

- Translation prompts are restructured so the English content forms an identical prefix across all target locales, which lets Gemini implicit context caching discount repeated input tokens when one page is translated into many languages
- Model thinking is turned down to the minimum for translation work, where it added cost and latency without quality gains. Thinking tokens are now also counted in the reported usage, so cost estimates match what Google actually bills
- Batch results are priced at the 50% batch rate in the cost report; `--concurrency` now applies to direct mode only

## Fixed

- `scribe translate` no longer crashes with "no such table: meta" when writing to a brand-new store file
